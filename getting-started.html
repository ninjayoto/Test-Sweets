What is Test Sweets?

Test sweets is a tool I created for myself to quickly setup a test suite to test Front-end Javascript. It takes the thinking out of the boilerplate code and can drop into an existing project to build an automated test suite. It is also the result of lots of my own research and failures, and I hope if you decide to use it will save you time, frustration, and hopefully make you a better programmer.

Benefits:
  • Constant immediate feedback on the state of your application
  • An early warning system that catches the worst kind of bugs: silent failures
  • A greater sense of confidence when making changes to your codebase
  • A goal in life.

Interestingly, learning about and implementing unit tests greatly improved my knowledge and confidence with Javascript. It inspires a new way of thinking about Javascript, and requires a strong understanding of the underlying application.

What Test Sweets is not

Test sweets is not a testing framework. It uses a test framework called qUnit, made by the team that makes jQuery.

There are other testing frameworks available like Mocha, Jasmine, or Buster. Test Sweets is compatible with those frameworks, but not out of the box.

Features:

  • List of sample assertions available in the qUnit test framework
  • A sample test runner for rapid cross browser testing
  • Configurations for integrating Grunt
  • Testing at the command line using PhantomJS

This guide will help you setup Test Sweets in your project and cover larger topics about Unit testing.

Philosophy

Testing front-end code in web applications is not as straight forward as one would assume. It is very possible to write poor, brittle tests, that cost more time to maintain than any advantages gained from automated testing.

Just like how writing clean and effective code adheres to concepts like "Separation of Concerns" and "Don't Repeat Yourself", your testing philosophy will determine the quality of your test suite.

Testing Principles

Every test must be atomic.

Every test you write should be able to run individually and in any order. A test should not rudely affect any other test. One way to achieve this is to append any markup required for the test to the fixture element provided in the test runner tests/runnrers/main.html

<div id="qunit-fixture"></div>

which will be magically wiped clean after every test, while waiting dutifully for the next test to run.

A test should be fast.

Think milliseconds fast. Long winded tests that take several seconds (or worse, minutes) to complete will deter your fellow developers from running the tests on every compile. Speed can be achieved through mocking connections to remote servers, and overriding animation timers to 0 seconds.

Write once, run forever

Tests should be robust and run obediently until they have something useful to report. They should not be tightly coupled, brittle pieces of code that require constant attention.
